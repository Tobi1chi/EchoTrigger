# 事件触发音频回放代理

English version: [event_triggered_audio_replay_agent.md](event_triggered_audio_replay_agent.md)

   ↓  
ESP32-S3 节点  
   - I2S 采集  
   - PCM 分帧

   - 决策

## 摘要

这份文档记录的是一个相对务实的家庭 AI 助手方案：不做全时转写，而是基于短时音频回放实现事件感知。

核心设计是让家庭 agent 获得一双带上下文的“耳朵”，但又不把整个家庭变成一个长期监听、长期存档的系统。麦克风节点持续采集音频，并上传到中央 PC Hub。PC 端只保留一个短时滚动缓存，例如最近 10 分钟，而不是默认长期保存全部音频。系统也不会持续转写所有内容，而是等待真正有意义的家庭事件，例如开门、包裹到达或移动触发。事件发生后，再按需提取该事件前后的一小段音频窗口，并对这段窗口做分析。

这是一种“事件感知式听觉”，而不是“永远在听”的流水线。

## 设计动机

典型的家庭 AI 助手很容易做成一个表面可用的版本：

- 常开麦克风
- 唤醒词
- 语音转文字
- 简单 LLM 编排
- Home Assistant 工具调用

这套堆栈当然能用，但也很普通，本质上仍然是一个加了家庭自动化能力的语音助手。

这个项目想解决的是更有价值的问题：

- 比常开转写有更低的隐私成本
- 比全时 ASR 有更低的算力成本
- 比普通语音助手有更强的上下文理解能力
- 和真实家庭事件更好地对齐

目标不是“听见所有事”，而是“只在上下文说明值得听的时候再听”。

## 核心思路

家中的每个麦克风节点持续采集音频，并通过 UDP 上传到中央 PC Hub。PC Hub 维护一个覆盖式的循环缓存，只保留有限的近期窗口，例如最近 10 分钟。默认不建立长期音频档案。

当 Home Assistant 里发生相关事件时，系统把这个事件作为触发源，并执行：

1. 确定受影响的区域或设备
2. 从 PC Hub 的 Ring Buffer 中取出离该区域最近麦克风节点的事件前后音频窗口
3. 对短音频段做分析
4. 把结果转成结构化观察
5. 由 agent 决定是否通知、记忆、推断或执行动作

它在思路上很像游戏里的“即时回放”，只是对象换成了家庭里的音频上下文。

## 示例场景

一个很具体的例子是外卖到门口的场景。

1. 前门门锁打开，或前门门磁状态变化。
2. Home Assistant 发出一个前门互动开始的事件。
3. 系统从前门附近的麦克风节点提取音频。
4. 它截取一个回放窗口，例如：
   - 事件前 30 秒
   - 事件后 90 秒
5. 然后分析这段窗口中的：
   - 语音活动
   - 语音转写
   - 像 “delivery”“package”“left at the door”“thank you” 这样的关键词
   - 可选的非语音事件，比如敲门、脚步声、逗留
6. Agent 输出一个结构化解释。
7. 系统接着可以：
   - 通知用户外卖到了
   - 记录一条 memory
   - 如果没有有价值内容则忽略
   - 如果有人在门口停留过久则标记风险

所以这个系统不是单纯“听声音”，而是把声音和家庭上下文对齐起来。

## 为什么这样做有价值

### 隐私

这个设计避免了默认长期保存家庭对话。音频只会暂时保留在 Ring Buffer 中，只有在相关上下文存在时才会触发分析。

这比无差别的全时转写更容易在隐私边界上站得住脚。

### 算力效率

相比简单缓存和 VAD，语音识别的成本高得多。只在有意义的事件前后分析短窗口，可以避免把算力浪费在大量无关背景噪声上。

### 更强的上下文理解能力

普通助手通常只对直接命令做反应。这个设计让 agent 可以理解短时情境，例如：

- 外卖交付
- 门口对话
- 走廊里的异常争吵
- 另一个房间里小孩哭闹
- 某个运动触发之后的玻璃破碎声

也就是说，agent 得到的是一种有选择的感知能力，而不是盲目的持续转写。

## 架构拆解

系统可以拆成四层。

### 1. 麦克风节点层

家中每个房间或区域都可以部署一个基于 ESP32-S3 和 I2S 麦克风的小音频节点。

职责：

- 持续采集音频
- 把 PCM 音频打包成固定大小帧
- 持续通过 UDP 上行到 PC Hub
- 通过 MQTT 暴露控制和遥测
- 可选做轻量本地统计或预处理

推荐原则：

- 节点本身尽量简单且实时安全
- 不在节点上保存长时间音频
- 把 UDP 当作音频数据面
- 把 MQTT 当作控制和状态面

可能的硬件：

- ESP32-S3
- 像 INMP441 这样的 I2S 数字麦克风
- 可选带 PSRAM 的板卡，为未来更大缓冲或边缘预处理留空间

### 2. PC Audio Hub 层

这一层负责接收实时音频流，并把它变成短时可回放的上下文。

职责：

- 接收来自一个或多个麦克风节点的 UDP 音频包
- 根据序列号和时间戳检测乱序或丢包
- 为每个节点维护滚动 Ring Buffer，例如 10 分钟
- 按需提取回放窗口
- 可选把临时分析输入落成 WAV 或 PCM 文件

这一层是系统的短时记忆，不把长期存储压力压到嵌入式节点上。

### 3. 事件触发层

这一层监听来自 Home Assistant 和其他传感器的结构化事件。

可能的触发源：

- 门被打开
- 锁被解锁
- 门铃按下
- 包裹检测
- 人体移动
- mmWave 存在事件
- 报警状态变化
- 异常的传感器组合

每个触发最终都会生成一个分析请求，其中包括：

- 事件类型
- 来源设备
- 位置
- 起始时间
- 回放窗口定义
- 优先级或风险等级

触发层不直接处理音频，它的职责是：当上下文说明值得听时，向 PC Hub 请求一个时间窗口。

### 4. 音频分析层

这一层只处理被提取出来的短回放片段。

可能的处理步骤：

1. VAD，定位语音段
2. ASR，获得转写
3. 关键词检测
4. 非语音音频事件检测
5. 可选说话人识别
6. 结构化摘要

这一层应该输出适合机器消费的结构化观察，而不只是原始文本。

示例输出：

```json
{
  "event": "front_door_opened",
  "window_start": "2026-03-16T12:01:04Z",
  "window_end": "2026-03-16T12:03:04Z",
  "speech_detected": true,
  "transcript": "Your delivery is here. I left it at the door.",
  "keywords": ["delivery", "left at the door"],
  "audio_events": ["knock", "footsteps"],
  "confidence": 0.88,
  "summary": "Likely package delivery interaction at the front door."
}
```

### 5. Agent 决策层

像 OpenClaw 这样的 agent 运行时可以消费上面的结构化结果，并决定下一步动作。

可能动作：

- 推送通知
- 更新 memory
- 提出追问
- 触发 Home Assistant 动作
- 忽略低价值事件
- 升级高风险或异常事件

这一步才是音频理解真正转化为“助手行为”的地方。

## 通信模型

这个通信模型有意拆成两条面。

### UDP 数据面

UDP 用来承载从 ESP32-S3 节点到 PC Hub 的连续音频流。

推荐初版格式：

- 16 kHz
- 16-bit PCM
- mono
- 固定包时长，例如 20 ms

每个包应包含：

- 节点标识
- 包序号
- 采集时间戳
- 采样率和格式元信息
- PCM payload

这样可以让音频链路保持简单且低延迟。
